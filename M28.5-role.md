## 0. Context & Position in the Roadmap

### Why this module exists

M11 (Dashboard Monitoring) is complete but uses a single payload and a single layout for every role. The result:

- The treasurer sees a Task Kanban widget that is irrelevant to her work
- The secretary sees a finance summary that is not her responsibility
- A regular member sees an approval queue he cannot act on
- A project lead sees org-wide data when he only manages one proker

This is not a styling issue — it is **information noise** that makes users unable to identify what they need to act on. This module rebuilds the dashboard experience and sidebar navigation so every role receives a focused, actionable, noise-free view.

## 1. Role Definitions & Hierarchy

Prokerin uses Spatie Laravel Permission. Roles are defined in `AGENTS.md` section 10. Below is the full hierarchy with business context relevant to dashboard behavior.

### 1.1 Platform Level (excluded from workspace dashboard)

| Role | Scope | Dashboard |
|------|-------|-----------|
| `super_admin` | Platform-wide | Filament `/internal-admin` only |
| `campus_admin` | Campus-wide (M24) | Separate read-only panel — not built yet |

These roles do **not** receive a workspace dashboard. This module does not touch them.

### 1.2 Organization Level

| Role | Indonesian Org Equivalent | Data Scope | Approval Authority |
|------|--------------------------|------------|-------------------|
| `organization_owner` | Ketua Umum | Entire organization | Approve proposal, LPJ, finance |
| `organization_admin` | Wakil Ketua / Sekjend | Entire organization | Approve proposal, LPJ (delegated) |
| `secretary` | Sekretaris | Documents, meetings, proposals | No finance approval |
| `treasurer` | Bendahara | Finance, RAB, realization | Approve financial transactions |
| `project_lead` | Ketua Pelaksana | Own proker(s) only | Approve tasks in own proker |
| `division_coordinator` | Koordinator Divisi | Own division's proker | No formal approval |
| `member` | Anggota Aktif | Assigned tasks and joined proker | None |
| `viewer` | Alumni / Pembina / Observer | Read-only public org data | None |

### 1.3 Multi-Role Resolution Rule

A user can hold multiple roles in the same organization. Dashboard variant is determined by the **highest role** the user holds in the currently active organization.

Resolution hierarchy (highest → lowest):

```
organization_owner
  → organization_admin
    → secretary
      → treasurer
        → project_lead
          → division_coordinator
            → member
              → viewer
```

`project_lead` and `division_coordinator` share the same dashboard variant (Operational Dashboard) because their information needs are substantially overlapping.

## 2. Five Dashboard Variants

### Variant A — Pimpinan Dashboard
**For:** `organization_owner`, `organization_admin`

**Philosophy:** Leaders need a command center — one place to see the health of the entire organization and take approval decisions without navigating away.

**Widget layout:**

```
┌─────────────────────────────────────────────────────────┐
│ ROW 1 — KPI Cards (4 columns)                           │
│  [Proker Aktif]  [Rata-rata Progress]  [Total Anggota]  │
│  [Sisa Anggaran (Rp)]                                   │
├─────────────────────────────────────────────────────────┤
│ ROW 2 — Approval Queue (full width)                     │
│  List: Proposal pending | LPJ pending | Finance pending │
│  Inline Approve / Minta Revisi buttons per item         │
├──────────────────────┬──────────────────────────────────┤
│ ROW 3 LEFT           │ ROW 3 RIGHT                      │
│ Proker Prioritas     │ Aktivitas Anggota                │
│ (top 3 by nearest    │ (most active members,            │
│ deadline + progress) │ members with no activity         │
│                      │ in last 7 days)                  │
├──────────────────────┼──────────────────────────────────┤
│ ROW 4 LEFT           │ ROW 4 RIGHT                      │
│ Ringkasan Keuangan   │ Agenda Rapat Mendatang           │
│ (RAB vs Realisasi    │ (next 3 meetings,                │
│ all proker,          │ inline "Buat Rapat" button)      │
│ horizontal bar chart)│                                  │
└──────────────────────┴──────────────────────────────────┘
```

**Quick action buttons (rendered in dashboard page header):**
- "Buat Proker Baru" → `/proker/create`
- "Undang Anggota" → `/members/invite`
- "Lihat Semua Approval" → `/approvals`

**Data fetched by `PimpinanDashboardPayloadAction`:**

```php
return [
    'kpiMetrics'       => [...],  // proker count, avg progress, member count, remaining budget
    'approvalQueue'    => [...],  // proposal + LPJ + transactions with status submitted/review_submitted
    'priorityProjects' => [...],  // top 3 proker by nearest deadline, with progress_percentage
    'financeSummary'   => [...],  // per-proker RAB vs approved realisasi
    'upcomingMeetings' => [...],  // next 3 meetings from meetings table
    'memberActivity'   => [...],  // last_activity_at per member (from last task update or login)
];
```

### Variant D — Operasional Dashboard
**For:** `project_lead`, `division_coordinator`

**Philosophy:** Operational leaders only care about the proker and team they directly manage. Org-wide data is noise for them.

**Widget layout:**

```
┌─────────────────────────────────────────────────────────┐
│ ROW 1 — My Proker KPI Cards (4 columns)                 │

├──────────────────────┬──────────────────────────────────┤
│ ROW 2 LEFT           │ ROW 2 RIGHT                      │
│ Proker Saya          │ Task Perlu Perhatian             │
│ (card per proker,    │ (overdue or due in next 3 days,  │
│ progress bar,        │ with PIC name)                   │
│ status, deadline)    │                                  │
├──────────────────────┼──────────────────────────────────┤
│ ROW 3 LEFT           │ ROW 3 RIGHT                      │
│ Deadline Mendatang   │ Ringkasan Tim                    │
│ (mini timeline:      │ (per member: done tasks vs       │
│ proposal due, LPJ    │ open tasks in this proker)       │
│ due, event date)     │                                  │
└──────────────────────┴──────────────────────────────────┘
```

**Quick action buttons:**
- "Tambah Task" → `/timeline/create`
- "Update Status Proker" → `/proker/{slug}/edit`
- "Lihat Kanban Tim" → `/timeline`

**Scoping rules:**
- `project_lead`: all widgets show data only from proker where `project_members.user_id = auth()->id()` and `project_members.role = 'project_lead'`
- `division_coordinator`: show proker linked to this user's division; fallback to proker where user is a member if no division relation exists
- If user is project_lead on multiple proker, show data aggregated across all of them with per-proker breakdown

**Data fetched by `OperasionalDashboardPayloadAction`:**

```php
return [
    'kpiMetrics'         => [...],  // proker count, task done/total, overdue count, remaining budget
    'myProjects'         => [...],  // proker where user is project_lead or division member
    'urgentTasks'        => [...],  // tasks with due_date <= now+3days, not completed, with PIC
    'upcomingMilestones' => [...],  // proposal due, LPJ due, event date from user's proker
    'teamSummary'        => [...],  // per proker, per member: done vs open task count
];
```

```typescript
export type DashboardVariant =
  | 'pimpinan'
  | 'sekretaris'
  | 'bendahara'
  | 'operasional'
  | 'member'
  | 'viewer';

export interface SidebarMenuItem {
  label: string;        // Bahasa Indonesia — e.g. "Proker", "RAB & Keuangan"
  href: string;
  icon: string;         // lucide-react icon name

  groupLabel: string;   // Bahasa Indonesia — e.g. "Utama", "Administrasi"
  items: SidebarMenuItem[];
}

export type RoleMenuMap = Record<DashboardVariant, SidebarMenuGroup[]>;
```

## 4. Design System Constraints

**This module must visually match every existing page in the app without exception. No new design patterns, no new color schemes.**

Reference `AGENTS.md` section 8 for the full styling rulebook. Key constraints:

### Color Tokens

```
Primary green    : #24695c   → buttons, active nav, primary badges
Secondary brown  : #ba895d   → accent, secondary actions
Success dark     : #1b4c43   → success status badges
Danger red       : #d22d3d   → error, failed, overdue states
Page background  : #f5f7fb   → body bg
Card background  : #ffffff   → all card bg
Border           : #e6edef   → all card and table borders
Body text        : #242934
Muted text       : #59667a  or  #717171
```

### Components — always reuse, never recreate

| Need | Component |
|------|-----------|
| Page wrapper with stat row | `ModuleOverview` |
| Data table with sorting | `VihoDataTable` |
| Status chip | `VihoStatusBadge` |
| Card container | `VihoCard` |
| Flash success / error | `FlashBanner` (already in AppLayout) |
| Form inputs | shadcn/ui `Input`, `Select`, `Switch` from `@/Components/ui/` |
| Buttons | shadcn/ui `Button` with Viho color overrides via `cn()` |
| Charts | Use existing chart library already in the project — do not add a new one |

### Layout rules

- Stat cards: 4-column on desktop (`lg:grid-cols-4`), 2-column on tablet (`sm:grid-cols-2`), 1-column on mobile
- Widget grid: 2-column on desktop (`lg:grid-cols-2`), 1-column on mobile
- Full-width widgets span both columns
- Page header: title (H1 in Indonesian) + breadcrumb + quick action buttons — same pattern as `/meetings` and `/certificates`
- All spacing uses Tailwind default scale only — no custom values

### 5.3 `DashboardRoleResolverAction`

```
Responsibility:
- Accept: authenticated User + active Organization
- Query user's Spatie roles scoped to the given organization
- Apply the resolution hierarchy defined in section 1.3
- Return: DashboardVariant enum case
- This action is SECURITY CRITICAL — it must always resolve from server-side Spatie data,
  never from client-submitted values
```

### 5.4 `DashboardPayloadAction`

```
Responsibility:
- Accept: User, Organization, DashboardVariant
- Dispatch to the correct variant sub-action
- Return: array payload ready for Inertia::render()
- No business logic here — pure dispatcher
```

### 5.5 Variant Sub-Actions (one file each)

Rules that apply to ALL sub-actions without exception:

```
1. Every single query MUST include: ->where('organization_id', $org->id)
   No exceptions. This is the primary multi-tenancy guard.

2. Use eager loading for all relationships. Never trigger N+1.
   Correct: Project::with(['tasks', 'members'])->where(...)
   Wrong  : foreach ($projects as $p) { $p->tasks; }

3. All numeric aggregations (totals, percentages, counts) must use
   database-level aggregation (sum(), count(), avg()) — not PHP loops.

4. Cache expensive calculations in Redis:
   Key pattern : dashboard:{variant}:{org_id}:{user_id}
   TTL         : 5 minutes (300 seconds)
   Invalidate  : when relevant data changes (use model observers or explicit cache clear in actions)

5. Return plain arrays to Inertia — never raw Eloquent models or collections.
   Use array casting or explicit array mapping.
```

### 5.6 `SidebarMenuAction`

```
Responsibility:
- Accept: User, Organization
- Resolve DashboardVariant via DashboardRoleResolverAction
- Return the correct menu array for that variant
- Compute badge counts for the resolved role:
    approval_count        → count of proposals + LPJ + transactions with
                           status in ['submitted', 'review_submitted', 'pending_approval']
                           scoped to org; only included for roles that have approval access
    pending_tasks         → count of tasks assigned to this user that are overdue
                           or due today, not completed
    unread_notifications  → count from notifications table where notifiable_id = user->id
                           and read_at IS NULL

- Cache badge counts in Redis with TTL: 60 seconds
  Key: sidebar_badges:{org_id}:{user_id}
- Return structure:
  [
    ['groupLabel' => 'Utama', 'items' => [
      ['label' => 'Dashboard', 'href' => '/dashboard', 'icon' => 'LayoutDashboard', 'badgeCount' => null],
      ['label' => 'Notifikasi', 'href' => '/notifications', 'icon' => 'Bell', 'badgeCount' => 3],
    ]],
    ...
  ]
```

## 8. Security Rules

The following rules are non-negotiable and must be enforced in every action and controller:

```
RULE 1 — TENANT ISOLATION
Every query touching org data MUST include: ->where('organization_id', $org->id)
No exceptions. The test suite must verify this for every variant sub-action.

RULE 2 — SERVER-SIDE ROLE RESOLUTION ONLY
DashboardVariant is resolved exclusively from Spatie permission data on the server.
Never read dashboardVariant from the request, query string, or any client input.

RULE 3 — CLEAN INERTIA PAYLOAD
Never return raw Eloquent models to Inertia.
Always return plain PHP arrays. Use explicit array mapping or API Resources.
Never include columns the frontend does not need (password hashes, raw S3 paths, tokens).

RULE 4 — SIDEBAR IS NOT THE ONLY GUARD
Hiding a menu item from the sidebar does not protect the route.
Every controller method must still call $this->authorize() or check Policy.
The sidebar filter is a UX convenience, not a security boundary.

RULE 5 — BADGE COUNTS ARE SCOPED
Badge count queries (approval_count, pending_tasks) must only count items
that the requesting user's role is actually authorized to see and act on.
Never return a count of items the user cannot access.
```
